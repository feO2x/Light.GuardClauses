# Contributing

Thanks for helping improve Light.GuardClauses.

1. Open an issue before starting a new assertion or substantial change so its API and scope can be discussed.
2. Keep each pull request focused on one change and follow the style of the surrounding code.
3. Add functional tests for behavior changes. Add or update a benchmark when performance is a relevant risk.
4. Build and test the complete solution before submitting the pull request.

The [contributor and build guide](docs/contributing-and-building.md) describes the required SDK, repository layout, build commands, test-framework overrides, assertion conventions, analyzers, benchmarks, and source-export projects.

User-facing guidance belongs under [`docs`](docs/README.md). Keep the root [README](README.md) concise and update documentation in the same pull request as the behavior it describes.
