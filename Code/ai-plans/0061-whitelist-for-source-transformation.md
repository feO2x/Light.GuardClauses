# Issue 61 - Whitelist for Source Transformation

## Rationale

`Light.GuardClauses.SourceCodeTransformation` merges all `Check`/`Throw` source files into a single file for consumers who want to embed Light.GuardClauses as source code instead of referencing the DLL. Today, the merger always includes every assertion. Some consumers only need a handful of assertions and want a smaller output file. We introduce an opt-in whitelist, expressed as a dedicated `AssertionWhitelist` sub-object on `SourceFileMergeOptions` with one entry per assertion, so callers can select exactly the assertions they need without cluttering the top-level options record.

A file-name-based approach to trimming the `Throw`/`FrameworkExtensions`/`Exceptions` side turns out not to work: of the 100 current assertions, only 14 have a same-named `Throw.<Name>.cs` file, and several `Throw.*.cs` files bundle unrelated methods that each serve a different, unrelated assertion (e.g. `Throw.Uri.cs` contains `MustBeAbsoluteUri`, `MustBeRelativeUri`, `UriMustHaveScheme`, and `UriMustHaveOneSchemeOf`, each called by a different `Check.*.cs` file). Getting genuinely minimal output therefore requires knowing, per method, what's actually called - not per file. We do this with a Roslyn semantic reachability analysis: starting from the `Check` methods the user selected, resolve every outgoing call to its declaring symbol and recurse, producing the exact set of `Throw`/`FrameworkExtensions`/`Exceptions`/`Check` members actually needed. This also naturally solves the original cross-assertion dependency problem (e.g. `MustBeGreaterThan` calling `MustNotBeNullReference`) - a required assertion is pulled in automatically even if the user didn't select it, because it's simply part of the reachable set.

As a safety net for anything the semantic analysis might still miss (ambiguous overload resolution, unusual call shapes), the tool additionally compiles the resulting project after merging and warns - without withholding the generated file - if that build fails. Giving each assertion its own dedicated entry also gives us a natural home for a second, independent refinement: today's `RemoveOverloadsWithExceptionFactory` option is all-or-nothing across the whole library, but consumers may want to keep the exception-factory overload for some assertions while dropping it for others - each assertion's entry gets its own `IncludeExceptionFactoryOverload` flag for that. When the whitelist isn't engaged (the default), behavior and output stay exactly as they are today.

## Acceptance Criteria

- [ ] `SourceFileMergeOptions` gets exactly one new property: `public AssertionWhitelist AssertionWhitelist { get; init; } = new();` - no other new properties are added directly to `SourceFileMergeOptions`.
- [ ] A new `AssertionWhitelist` record (its own file, e.g. `AssertionWhitelist.cs`) exposes `bool IsEnabled { get; init; } = false;` (the mode toggle - when `false`, no assertion is filtered and output is unchanged from today) plus one property per assertion (named after the `<Name>` portion of its `Check.<Name>.cs` file), each typed as a nested `AssertionEntry` record.
- [ ] `AssertionEntry` is a small record with `bool Include { get; init; } = true;` and `bool IncludeExceptionFactoryOverload { get; init; } = true;`.
- [ ] When `AssertionWhitelist.IsEnabled` is `true`, a new Roslyn-based member reachability analyzer computes, starting from the `Check` methods belonging to assertions with `Include == true` (minus any member removed by `IncludeExceptionFactoryOverload == false`), the full transitive set of members actually invoked - across `Check` (pulling in required-but-unselected assertions automatically), `Throw`, `FrameworkExtensions`, and `Exceptions`.
- [ ] `SourceFileMerger` uses this reachable set to decide, at member level, what survives in the final `Throw`/`FrameworkExtensions`/`Exceptions` types, instead of including those files/members unconditionally. `Check.cs`/`Throw.cs` (the partial class shells) and the attribute files governed by their own existing options are untouched by this filtering.
- [ ] The existing exception-factory-overload removal step in `SourceFileMerger` (currently governed solely by the global `RemoveOverloadsWithExceptionFactory` flag) additionally removes a given assertion's exception-factory overload whenever that assertion's `IncludeExceptionFactoryOverload` is `false`, independent of `AssertionWhitelist.IsEnabled` - and this filtering happens *before* reachability analysis runs, so a deliberately-excluded overload's own outgoing calls don't keep unrelated code alive.
- [ ] After the target file is written, if a project file (`.csproj`) is found in the same directory as `options.TargetFile`, the tool builds that project (via `dotnet build`) to validate that the generated file compiles. This runs regardless of `AssertionWhitelist.IsEnabled`, as a safety net independent of the reachability analysis above. If `options.TargetFile` doesn't sit inside a project directory, the validation step is skipped.
- [ ] If that build fails, the tool prints a clear warning (including the captured build output) and the process exits with a non-zero code, while the generated file is still left on disk so the user can inspect or fix it.
- [ ] `Code/Light.GuardClauses.SourceCodeTransformation/settings.json` is removed from `.gitignore` and a committed version is added with `AssertionWhitelist.IsEnabled: true` and every assertion entry's `Include` / `IncludeExceptionFactoryOverload` set to `true` (functionally identical output to today, but ready for users to flip flags - or search-and-replace `true` to `false` and back for the ones they want - to customize).
- [ ] `.gitignore` instead excludes a new, optional `Code/Light.GuardClauses.SourceCodeTransformation/settings.local.json`. `Program.cs` loads it after `settings.json` (and before `AddCommandLine`), so developers can keep personal overrides (e.g. a local `SourceFolder`/`TargetFile`, or personal whitelist tweaks) without modifying the committed template.
- [ ] Automated tests need to be written.

## Technical Details

- **Options shape**: `AssertionWhitelist` (new file, `Light.GuardClauses.SourceCodeTransformation/AssertionWhitelist.cs`) is a sealed record holding `IsEnabled` plus one `AssertionEntry`-typed property per assertion, all defaulting to `new()` (i.e. `Include = true`, `IncludeExceptionFactoryOverload = true`). Keeping every default `true` means the record's built-in defaults, the committed `settings.json` template, and "include everything, unmodified" all agree - only `IsEnabled` needs to flip for filtering to kick in, and only individual `Include`/`IncludeExceptionFactoryOverload` flags need to flip for someone to actually customize the output. Reflection maps each assertion name to its `AssertionEntry` property, so nothing needs a second hand-maintained list of assertion names.

- **Member reachability analyzer** (the reinstated dependency resolver, now semantic rather than file-name-based): add a new class, e.g. `MemberReachabilityAnalyzer`, in `Light.GuardClauses.SourceCodeTransformation`. This is the part of the plan that needs Roslyn's semantic layer, not just its syntax layer:
  - **Compilation setup**: parse every source file with `CSharpSyntaxTree.ParseText(text, csharpParseOptions, path: file.FullName)` - critically, passing `path:` this time (the current merger doesn't), so a resolved symbol's declaring location can be traced back to its originating file. Build one `CSharpCompilation` from all of these trees. Assemble a reference set using the running process's trusted platform assemblies (`AppContext.GetData("TRUSTED_PLATFORM_ASSEMBLIES")`, split and turned into `MetadataReference.CreateFromFile(...)` calls) plus any package references the library needs (e.g. `System.Collections.Immutable`) - this compilation only needs to be good enough for symbol resolution to succeed on the library's own calls, not diagnostically clean, so it doesn't need to perfectly replicate `netstandard2.0`.
  - **Seed set**: the final `Check` members surviving `Include`/`IncludeExceptionFactoryOverload` filtering (see above) - i.e. exactly what the user asked to keep.
  - **Traversal**: for each member in the current frontier, get its `SemanticModel` (`compilation.GetSemanticModel(tree)`) and walk its body for `InvocationExpressionSyntax`/`MemberAccessExpressionSyntax`/`ObjectCreationExpressionSyntax` nodes. Resolve each via `semanticModel.GetSymbolInfo(node)`; normalize the resulting symbol with `.OriginalDefinition` (collapses generic instantiations back to the declaring method/constructor). If `.Symbol` is `null` because overload resolution couldn't fully bind, fall back to `.CandidateSymbols` and treat all of them as reachable rather than guessing - consistent with preferring over-inclusion over a broken build.
  - **Mapping back to source**: use `symbol.DeclaringSyntaxReferences` to get to the declaring `MethodDeclarationSyntax`/`ConstructorDeclarationSyntax`/etc., and from there `.SyntaxTree.FilePath` to identify the originating file/member. Add newly-discovered members to the frontier and repeat (BFS/worklist) until no new members are found.
  - **Result**: the full reachable member set, spanning `Check` (any additional assertions pulled in transitively, even with `Include == false`), `Throw`, `FrameworkExtensions`, and `Exceptions`.

- **`SourceFileMerger.CreateSingleSourceFile`**: when `options.AssertionWhitelist.IsEnabled` is `true`, run the reachability analyzer once, after assembling the filtered `Check`/`Throw` member lists (`Include` + `IncludeExceptionFactoryOverload` applied) but before final assembly. When building the final `Throw`, `FrameworkExtensions`, and `Exceptions` types, intersect their members with the reachable set (same `.Members.Where(...).WithMembers(...)` pattern already used for the `RemoveOverloadsWithExceptionFactory` filter, just driven by set membership instead of a parameter-name check). Any `Check` member pulled in only via reachability (not explicitly `Include == true`) is added back into the final `Check` class the same way. `Check.cs`/`Throw.cs` shells and attribute files stay governed by their existing, separate options and aren't touched by this filtering.

- **Exception-factory overload trimming**: while merging, track which assertion name each `Check`/`Throw` member came from (the file-by-file merge loop already visits one file at a time, so this is a matter of recording the current assertion name alongside each member it contributes). The existing block around `options.RemoveOverloadsWithExceptionFactory` (which filters `Check`/`Throw` class members by checking for a parameter named `exceptionFactory`) is extended so a member is also removed when its recorded assertion's `IncludeExceptionFactoryOverload == false`. This runs before the reachability analyzer's seed set is built, so an explicitly-excluded overload's own calls (e.g. into `Throw.CustomException`) don't artificially keep that shared code alive on their own.

- **Build validation**: add a small class, e.g. `GeneratedFileBuildValidator`, in `Light.GuardClauses.SourceCodeTransformation`. After `SourceFileMerger.CreateSingleSourceFile` writes `options.TargetFile`, look for a single `.csproj` file in `Path.GetDirectoryName(options.TargetFile)` (this is how the default `Light.GuardClauses.Source.csproj` is found too - no name is hardcoded, since `TargetFile` is itself configurable). If none is found, skip validation - there's nothing to build. Otherwise run `dotnet build` against it via `System.Diagnostics.Process`, capturing stdout/stderr and the exit code, regardless of `options.AssertionWhitelist.IsEnabled`. On a non-zero exit code, write the captured output to the console as a warning and have `Program.Main` return a non-zero code - the merged file stays on disk either way.

- **Config layering**: `Program.cs` currently does `new ConfigurationBuilder().AddJsonFile("settings.json", true).AddCommandLine(args)`. Add `.AddJsonFile("settings.local.json", optional: true)` between those two calls, mirroring the common `appsettings.json` + `appsettings.Development.json` layering convention - command-line arguments still win over both files.

- **Committed `settings.json` maintenance**: this file must be regenerated/updated whenever an assertion property is added to or removed from `AssertionWhitelist`, similar to the other manual touch points already required when adding a new assertion (production file, tests, etc.).

- **Tests**: add to `Light.GuardClauses.SourceCodeTransformation.Tests`, following the existing pattern of parsing real files from the `Light.GuardClauses` source folder (see `ParseCSharpWithRoslynTests`). Cover: reachability analysis correctly pulls in a transitively-required assertion that wasn't explicitly selected (e.g. selecting only `MustBeGreaterThan` must still retain `MustNotBeNullReference`); reachability correctly trims an unrelated method out of a bundled `Throw.*.cs` file (e.g. selecting only `MustBeAbsoluteUri` must not retain `UriMustHaveScheme`); per-assertion exception-factory-overload trimming both with and without `AssertionWhitelist.IsEnabled`; that `AssertionWhitelist.IsEnabled = false` reproduces today's full output regardless of the per-assertion flags; and the build-validation step for a complete selection (build succeeds), a non-whitelisted merge that still targets a project directory (validation still runs), and a `TargetFile` outside any project directory (validation is skipped).

- No benchmarks needed - this feature isn't performance-sensitive (it runs once per source export, not in a hot path).
